# 협업 가이드

## 브랜치 전략

- `main`: 릴리스 가능한 안정된 코드
- `develop`: 개발 중인 코드
- `feature/*`: 기능 개발 (예: `feature/player-movement`)
- `fix/*`: 버그 수정 (예: `fix/jump-bug`)

## 커밋 메시지 규칙

```
[타입] 제목

상세 설명 (선택사항)

Closes #이슈번호
```

### 타입
- `feat`: 새로운 기능
- `fix`: 버그 수정
- `refactor`: 코드 리팩토링
- `docs`: 문서 수정
- `style`: 코드 스타일 (기능 변화 없음)
- `test`: 테스트 추가
- `chore`: 빌드/의존성 등 유지보수

## PR 프로세스

1. 자신의 브랜치에서 개발
2. 테스트 실행 및 확인
3. Pull Request 생성
4. 팀원의 코드 리뷰
5. 피드백 반영 후 병합

## 파일 구조 규칙

- 새로운 기능은 관련 폴더에 정리
- 음수 파일은 `assets/` 아래 분류
- 소스 코드는 `src/` 아래 정리

## 의사소통

- 주요 결정사항은 이슈에서 논의
- 급한 사항은 실시간 채팅으로 공유
